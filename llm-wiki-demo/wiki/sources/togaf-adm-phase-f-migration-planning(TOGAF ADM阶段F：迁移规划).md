# TOGAF ADM阶段F：迁移规划

> 证据范围：第10章，raw 页116–125；状态：`single-source`。

## 摘要(Summary)

阶段 F 把阶段 E 的路线图和计划草案整合到企业实际的变革投资组合中。它为每个工作包和项目定义 [[business-value(业务价值)]]，估算资本、运营、资源和时间要求，结合成本收益及风险进行排序，并依据能力增量和可吸收的变化节奏确认过渡架构。

本阶段完成正式的架构路线图和 Implementation and Migration Plan，把工作包分配到项目和投资组合，明确里程碑、依赖、成本、资源、收益及风险。完成后，治理责任从架构开发转向实现架构，经验教训进入阶段 H 改进企业架构能力。

![阶段F迁移规划](../../raw/C220-Part1e-Architecture%20Development%20Method_images/p00116_img001.png)

## 目的和目标

- 完成架构路线图和支持它的实施迁移计划。
- 与企业整体业务规划、项目组合及运营管理协调。
- 让关键利益相关者理解工作包和过渡架构的价值、成本及风险。

## 关键输入

阶段 E 的路线图和计划草案、工作包、过渡架构、实施因素、能力评估、四域架构、需求、变更请求及企业管理框架。

## 核心步骤

1. 协调业务规划、企业架构、项目组合和运营管理框架。
2. 定义价值尺度并为工作包和项目分配业务价值。
3. 估算资源、资本/运营成本、时间和交付载体。
4. 通过成本收益和风险评估排序迁移项目，并取得利益相关者共识。
5. 确认路线图、过渡架构和架构定义增量。
6. 完成项目、活动、依赖、资源及里程碑计划。
7. 结束本轮架构开发并记录经验教训。

## 技术、关键决策和跨阶段依赖

- 技术：业务价值评估、成本收益分析、风险评估、项目优先级排序、资源估算、投资组合依赖分析和路线图确认。
- 关键决策：哪些项目先做、如何融资和配置资源、何时达到各过渡状态，以及哪些工作延期、取消或移交后续循环。
- 跨阶段依赖：本阶段定稿阶段 E 的路线图和计划草案；获批计划、治理模型和架构定义交给阶段 G，经验教训和后续请求进入阶段 H。

## 主要输出

- 获批的 Implementation and Migration Plan。
- 项目/投资组合分解、工作包分配、里程碑、成本、资源、收益和风险。
- 最终架构定义、需求规格、过渡架构和架构路线图。
- 可复用 ABB、实施治理模型、下一轮架构工作请求及能力改进请求。

## 关键主张(Key Claims)

1. 迁移计划必须进入企业现有管理框架，不能作为架构团队的孤立计划存在。[原文，§10.3.1，raw 页119–120](../../raw/C220-Part1e-Architecture%20Development%20Method.md)
2. 业务价值需要先定义度量方式，再用于资源分配、延期或取消决策。[原文，§10.3.2，raw 页120–121](../../raw/C220-Part1e-Architecture%20Development%20Method.md)
3. 项目优先级取决于净收益、风险和战略依赖；即时收益小的项目仍可能是高价值路径上的关键前置项。[原文，§10.3.4，raw 页121](../../raw/C220-Part1e-Architecture%20Development%20Method.md)
4. 过渡架构是投资组合里程碑，路线图和计划的详细程度应与架构定义文档匹配。[原文，§10.3.5–10.3.7，raw 页121–123](../../raw/C220-Part1e-Architecture%20Development%20Method.md)
