# 摄取（ingest）工作流 - Reference

将新知识源引入wiki的详细操作指南。Schema文件包含了简化的内容，本文档涵盖了所有特殊情况及相关说明。

## 触发条件(trigger)

用户向 `raw/` 添加了一个文件，并指示 LLM 对其进行处理。使用的触发词包括：“摄取”、“处理”、“编译”、“增加这个文件”、“将`{file}`更新到wiki中”、“ ingested”、“process”、“add this source”、“read this and update the wiki”等。

## Step 0：预检

1. 按照 `references/config/extend-schema.md` 来设置 `EXTEND.md` 偏好文件。如果找不到偏好设置文件，请先执行初次偏好设置流程，然后再继续操作。
2. 确认文件存在于 `raw/`中。
3. 根据以下表格确定文件的类型：
   | 类型 | 处理方式 |
   | --- | --- |
   | Markdown / text | 直接阅读 |
   | PDF | 提取文本（询问用户是否有工具提供） |
   | Image | 描述内容，将信息以视觉形式呈现出来 |
   | CSV / JSON | 总结结构和主要数据点 |
   | Other | 询问用户如何进行处理 |
4. 检查 `wiki/log.md`，识别这个数据源之前是否已经被导入过？如果是，则提示用户：“该数据源已于{日期}被导入。是重新导入（更新）还是跳过此过程？”
5. 如果 `bm25.mode: auto_prompt` ，检查该wiki是否超过了配置的 BM25 阈值。如果超过了，请询问是否需要在导入数据之前初始化 BM25。如果用户同意，请执行 `references/workflows/bm25.md` 操作。

## Core Sequence

### step 1：读取并提炼候选内容

阅读整个源文件，在内存中生成一份笔记（不要保存为文件）并展示给用户：

- 原文摘要和整体理解
- 3-5 条关键主张
- 候选概念，以及建议的创建、更新、合并、拆分或忽略动作
- 候选实体（人物、组织、产品、地点等）、实体类型，以及建议的创建、更新或忽略动作
- 使用 `wiki/index.md`、现有页面和 BM25 候选结果识别已有匹配
- 任何与现有 wiki 知识重叠、矛盾或扩展现有知识的内容
- 为每个拟创建或更新的来源、实体、概念、比较、综合和概览页面确定最终文件名，格式为 `english-slug(中文标题).md`

此阶段只能读取和分析来源及现有 wiki，不得写入任何 wiki 派生页面。
明确询问用户是否确认该清单，以及是否需要修正摘要、关键内容、概念、实体或处理重点。

### 步骤 2：与用户进行讨论

请提出 2-3 个关于关键内容的简短总结。提问：

- “有什么是需要特别强调还是略去不谈的呢？”
- “需要对我之前的阅读结果进行任何修正吗？”

只有用户明确确认后，才能继续创建或更新来源页、概念页、实体页、概念表、索引、日志或概览。用户提出任何修正后，必须根据修正重新生成并展示完整确认清单，完成修正后再次确认；修正本身不视为授权。

如果用户说 `ingest silently`（直接摄取）或 `batch ingest`（批量摄取），可以跳过此确认门禁，但仍应在内部完成同等的摘要、概念和实体分析。

### 步骤 3：创建来源摘要页

确认通过后，撰写 `wiki/sources/{english-slug}({中文标题}).md`文档：

```yaml
---
title: "{english-slug}({中文标题})"
type: source-summary
created: { date }
updated: { date }
sources: [{ filename }]
tags: [{ auto-generated tags }]
---
```

注意：`sources` 只使用文件名（例如 `[{english-slug}({中文标题}).md]`），不要使用 `raw/{english-slug}({中文标题}).md` 这样的完整路径。

正文结构：

1. **摘要(Summary)**——2-3 段内容的概述
2. **关键主张(Key Claims)**——使用编号列表格式，并包含内嵌的引文（使用源文件的语言）
3. **提及的实体(Entities Mentioned)**——带 wikilink 的列表
4. **相关概念(Concepts)**——带 wikilink 的列表
5. **重要引文(Notable Quotes)**——原文引用（使用源文件的语言），并标明对应的页码／章节
6. **局限／偏见(Limitations / Bias)**——如果文章仅基于单一视角，且带有过时或偏颇的观点，请在此说明

### 步骤 4：关联更新

对每一个被该来源（source）所影响的实体（entity）和概念（concept）：

- 所有新页面（来源、实体、概念、比较、综合和概览）都必须使用 `english-slug(中文标题).md`。
- 英文 slug 使用小写 ASCII、连字符分隔且不含空格；中文部分使用页面正式标题。
- 如果目标文件已存在，则将其视为同一页面并更新；不得仅因中文标题变化创建重复页面。

- 如果 BM25 已启用，在创建新页面前，应先搜索现有的相关页面：

  ```bash
  python3 scripts/wiki_fts.py search "{candidate entity or concept}" --limit 10
  ```

- 如果页面已存在，则添加新信息，更新 `sources` frontmatter，如有必要，则修改**摘要(Summary)**。
- 如果页面不存在，则根据词来源中的信息创建页面。
- 如果来源与现有内容矛盾，则需要在两个页面中都添加冲突块(contradiction block)注释。

**冲突格式(Contradiction format)**：

```markdown
> ⚠️ 冲突（添加日期：{date}）
> {来源 A} 的主张：“{quote or paraphrase}”
> {来源 B} 的主张：“{quote or paraphrase}”
> 解决措施：{pending | 来源 B 优先（数据更新） | 两者在不同语境下均成立}
```

### 步骤 5：更新概念表(Update Concept Table)

在摄取过程中创建、重命名、合并、拆分、删除或大幅修改的概念，都更新到 `wiki/concept-table.md`中。

对每一行概念，保持这些字段为最新：

| 字段                                    | 维护规则                                                                             |
| --------------------------------------- | ------------------------------------------------------------------------------------ |
| `概念(Concept)`                         | 链接到 `wiki/concepts/` 下的持久化概念(Concept)页。                                  |
| `工作定义(Working definition)`          | 一个简洁、证据清晰的定义。                                                           |
| `在本 wiki 中的作用(Role in this wiki)` | 说明该概念为什么对当前知识库的作用。                                                 |
| `来源(Sources)`                         | 仅包含源文件名，与页面 frontmatter 风格一致。                                        |
| `相关页面(Related pages)`               | 重要的概念、实体、比较页或综合页。                                                   |
| `状态(Status)`                          | `high confidence`、`single-source`、`tentative`、`needs sources` 或 `contradicted`。 |
| `维护备注(Maintenance note)`            | 未来摄取或检查应当关注的事项。                                                       |

保持概念行按字母顺序排序。如果新来源改变了概念版图，也要更新“概念集群”小节。

### 步骤 6：更新索引(Update Index)

向 `wiki/index.md` 的“来源”表添加来源条目。添加或更新实体和概念条目；索引显示名使用`english-slug(中文标题)`，链接指向双名文件。

### 步骤 7：更新日志(Update Log)

将结构化条目附加到 `wiki/log.md`中。

### 步骤 8：修订概览(Revise Overview)

重新阅读 `wiki/overview(概览).md`，评估新的信息是否改变了整体的格局。如果是，则修订。如果不是，则跳过。即使是小型 wiki，也要保持 overview 最新，因为单个来源也可能重塑叙事。

### 步骤 9：刷新搜索索引

如果 BM25 已启用且 `auto_rebuild_after_ingest: true`：

```bash
python3 scripts/wiki_fts.py build
python3 scripts/wiki_fts.py stats
```

如果重建失败，不要回滚对 wiki 页面的编辑。将失败记录到 `wiki/log.md`，并在后续导航中回退到 `wiki/index.md` 加 `rg`。

## 批量摄取

当同时处理多个来源时：

1. 按顺序处理（不要并行，因为每个来源都可能影响下一个）
2. 默认跳过step 2逐个来源确认环节，但仍要提炼每个来源的摘要(summary)、候选概念(concepts)和实体(entities)；如果发现矛盾或其他高影响不确定性，暂停并请求用户确认。
3. 将日志条目合并为一个批量条目。
4. 处理好之后，统一修订一次 `wiki/concept-table.md` 和 `wiki/overview(概览).md`，不要在处理每个来源之后都修订。
5. 如果 BM25 已启用且配置为自动重建，批量处理好之后统一重建一次 BM25

## 特殊情况

| 场景                                     | 处理方式                                                                                                                                          |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| 来源与现有来源明显重叠                   | 记录重叠，只添加真正新增的信息。                                                                                                                  |
| 来源是"一手文档"（例如法律文本、规范）   | 总结内容，同时链接到 raw 文件以便查看准确的措辞                                                                                                   |
| 来源的语言不同于现有 wiki 页面           | 使用现有wiki页面的主要语言来编写此来源的 wiki 页面。wiki页面中的quote应使用来源语言。对于混合多种语言的跨来源页面，使用多数来源语言，或询问用户。 |
| 来源包含图片                             | 用文本描述重要图片的信息，并引用 `raw/assets/` 中的图片文件。                                                                                     |
| 来源非常长（书籍的章节、50页以上的材料） | 将来源分成几个部分来处理，为每个部分创建一个带有章节标题的摘要(summary)页面。                                                                     |
